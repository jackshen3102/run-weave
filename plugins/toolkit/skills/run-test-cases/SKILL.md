# 执行测试用例

核心原则：

> **用真实环境取证，一条一条来。测试用例是验收合约——只能改实现让它通过，不能偷改用例让它变绿。**

这个 skill 只负责**执行**用例。编写/补全用例由 `write-test-cases` 负责。

## 使用边界

适用：已有测试用例（`docs/testing/*-test-cases.md` 或用户直接给的），需要执行并验收。

不适用：用例还没写 → 先用 `write-test-cases`。

## 铁律（两种模式都遵守）

1. **真实取证，禁止用静态检查冒充。** `pnpm typecheck` / `pnpm lint` 只是前置门禁，**不能**作为 UI / 浏览器 / 行为用例的通过证据。
   - 浏览器页面路径：**必须**用 `$playwright-cli` 真实操作并取证（DOM / 截图 / 控制台 / 网络），不许用代码阅读、静态检查或想象代替。
   - 桌面端联动：用 `$computer-use` 把环境准备到目标状态，再用 `$playwright-cli` 做页面级验收。
   - 后端 / 协议 / CLI：跑用例文档里的 verify 脚本或临时 Node 脚本，给出命令和关键输出。
2. **不许偷改用例。** 严禁为了让测试通过而静默修改测试用例。要改用例，必须先停下、说明理由、由人工确认（判据见下）。
3. **不许跳步。** 用例文档写「任一失败即停」就必须停，不许跑完全部再汇总。
4. **逐条留证。** 每条用例给出：执行的命令 / 操作、关键输出或截图路径、通过或失败判定。承诺过浏览器验证却没执行，要明确写「未执行 + 阻塞原因」，不能糊弄过去。

## 两种执行模式

进入时先确认模式；用户没指定则用**模式 1（默认）**。

### 模式 1 · 严格逐条（默认）

- 按用例顺序逐条执行。
- **任一用例失败，立即停止**，不继续后面的 case。
- 报告：失败用例 ID、复现命令/操作、关键输出、初步判断（是实现 bug 还是用例本身问题），交人工决定下一步。
- 不主动改任何代码或用例。
- 适用：不确定用例本身是否可靠、或改动风险高时。防止「跑完再汇总 + 顺手改了不合理用例」把前面的验收全废掉。

### 模式 2 · 自愈（用户显式要求「失败就自己修」时）

- 逐条执行；用例失败时，**先判断失败归因**：
  - **实现 bug**（用例合理，代码不对）→ 修实现，重跑该用例直到通过，再继续下一条。修复可复用 `debugging-strategies`。
  - **用例本身有问题**（见下方判据）→ **停下**，说明理由，交人工确认，**不要**擅自改用例。
- 每次修实现都要重新取证，不能只改代码不重验。
- 全程仍受铁律约束：不静默改用例、浏览器路径必须真实取证。

## 「用例本身有问题」判据（命中即停，交人工）

出现以下任一情况，判定为用例问题而非实现问题，停下确认，不擅自修实现也不改用例：

- 用例的预期与「当前代码事实」或已确认的需求直接矛盾（需求已变更）。
- 用例前置条件已不成立（依赖的入口 / 接口 / 环境变量已不存在）。
- 用例预期本身自相矛盾，或无法取证（写得太模糊，无法判定通过）。
- 修复它需要改动明显超出本次需求范围的行为。

## 执行流程

1. 读用例文档，确认「必跑命令」和用例映射；确认执行模式。
2. 先跑前置门禁（typecheck / lint / verify 脚本），按文档顺序，任一失败即停。
3. 逐条执行用例，按模式规则处理失败。
4. 浏览器 / 桌面用例按铁律用 `$playwright-cli` / `$computer-use` 真实取证。
5. 需要可回放视频证据时，衔接 `recorded-browser-mcp-verification`。
6. 汇报：逐条结果、证据、停下点或已修项。

## 报告格式

- 模式与用例文档路径。
- 逐条：用例 ID → 通过/失败 → 命令或操作 → 关键证据（输出摘要 / 截图路径）。
- 停下点（模式 1 的首个失败 / 模式 2 判定的用例问题）：原因 + 复现 + 建议。
- 自愈模式额外列：修了哪些实现文件、重验结果。
