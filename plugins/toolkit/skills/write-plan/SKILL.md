# 编写实施计划

核心原则：

> **让执行者在实现方式上有空间，但不要让它在需求理解、边界和验收标准上有空间。**

计划要写清执行者不能猜的内容：目标、边界、约束、文件范围、接口、验证方式、验收标准、不做什么。它不是逐行代码生成器。

## 使用边界

适用：已有明确需求/目标，需要多步完成，且尚未进入实现（或用户要求先计划）。

不适用：

- 需求不清、连「做成什么样才算对」都判断不了 → 先补 spec 或提澄清问题；
- 单文件、低风险、几分钟能搞定的小修；
- 用户只要脑暴或开放式设计方向，不需要落地计划。

## 流程

1. **先查上下文。** 在已有代码库里写计划前，先读相关代码、文档、配置、测试，除非用户明确要求只基于需求草案。
2. **定范围。** 需求跨多个独立子系统时，拆成多个可独立验证的计划。
3. **选粒度。** 按风险选 L1/L2/L3，不要所有任务都顶格写。
4. **必须落盘。** 计划完成后保存为本地 Markdown 文件：用户指定路径则用指定路径；否则存到 `docs/plans/YYYY-MM-DD-<feature-name>.md`，目录不存在先创建。
5. **回复给路径。** 最终回复必须包含计划文件路径；仅在用户没说下一步时，简要给出执行方式建议。

## 计划粒度

按风险选择，不要过度精细：

- **L1 方向性**（低风险、执行者熟悉代码库）：目标、文件/模块范围、关键约束、验收与验证方式。不强制给代码。
- **L2 结构化**（中等复杂、多文件、需设计接口、交给 agent 执行）：任务拆分、精确文件路径、关键接口/类型、核心测试用例、验证命令、风险点。关键/易误解逻辑给代码，普通 glue code 引用现有模式。
- **L3 执行级**（高风险：认证/权限/支付/数据迁移/并发/跨平台/弱模型执行）：细粒度 checkbox 步骤、精确文件路径、关键或完整代码、精确命令与预期输出、逐任务验收清单、提交方式。

## 每个计划必须写清

- 目标、非目标（不做什么）；
- 用户可见行为、输入输出、业务规则；
- API 合约、数据结构、权限/安全边界、错误处理、兼容/迁移/回滚；
- 修改或新建的文件路径及各自职责；
- 验收标准 + 验证方式（命令/操作 + 预期输出 + 失败判断）；
- 高风险点：是否可能造成数据丢失、安全风险或线上回归。

业务逻辑、数据转换、权限、状态机优先给自动化验证；UI、配置、文档、一次性脚本可用手动验证清单。

涉及需求实现的计划，验收章节必须配套测试用例：用 `write-test-cases` 产出 `docs/testing/<主题>-test-cases.md`，并在计划里引用其路径，做到「计划完成 = 用例齐备」。执行阶段用 `run-test-cases`。

## 可以留给执行者

内部 helper 拆分、局部命名、非关键 glue code、与现有风格对齐的低风险实现。前提：目标、约束、文件位置、验收标准已明确。

## 落盘前自查

- 每个需求都对应到任务或非目标；
- 没有无法执行的占位（TBD/TODO、「加适当的错误处理」、「处理边界情况」等含糊表达）；
- 引用的类型/函数/字段/路径前后一致，且前文已定义；
- 粒度匹配风险，每个任务都有可执行的验证方式；
- 文件已写入计划路径。

!!! 必须使用中文编写计划文档
