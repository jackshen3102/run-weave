# 编写实施计划

使用本技能时，开头说明：

> 我正在使用 write-plan 技能来创建实施计划。

核心原则：

> **让模型在实现方式上有空间，但不要让它在需求理解、边界和验收标准上有空间。**

计划不是规格说明的替代品，也不是逐行代码生成器。计划要写清执行者不能猜的内容：目标、边界、约束、文件范围、接口、验证命令、验收标准和不做什么。

## 使用边界

使用本技能：

- 用户明确要求写 plan、方案、实施计划、执行计划、任务清单；
- 已有需求、规格说明、设计草案或明确目标；
- 任务需要多个步骤完成；
- 尚未进入实现，或用户明确要求先计划后实现；
- 计划需要交给 agent、subagent 或工程师执行。

不要直接使用：

- 需求还不清楚，连“做成什么样才算对”都无法判断；这时先补 spec 或提澄清问题；
- 只是单文件、低风险、几分钟能完成的小修；
- 用户只要脑暴、产品方向或开放式设计草案，不需要落地实施计划。

## 强制流程

1. **先查上下文。** 在已有代码库中写计划前，必须阅读相关代码、文档、配置或测试，除非用户明确要求只基于需求草案。
2. **先定范围。** 如果需求跨多个独立子系统，建议拆成多个可独立验证的计划。
3. **选择粒度。** 按风险选择 Level 1/2/3，不要所有任务都写成最高精度。
4. **必须落盘。** 计划完成后必须保存为 Markdown 文件。用户指定路径时按用户指定；否则保存到 `docs/plans/YYYY-MM-DD-<feature-name>.md`。如果目录不存在，先创建。
5. **最终回复给路径。** 最终回复必须包含计划文件路径，并只在用户没有明确下一步时提供执行方式选择。

如果这是 superpowers 风格工作流，且用户没有指定路径，可以保存到：

```text
docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md
```

## 计划粒度

### Level 1：方向性计划

适合低风险改动、简单功能、执行者熟悉代码库、用户希望保留实现空间。

必须包含：目标、文件或模块范围、关键约束、验收标准、测试和验证方式。不强制包含完整代码。

### Level 2：结构化实施计划

适合中等复杂功能、多文件变更、需要设计接口/数据结构、需要可靠测试、需要交给 agent 执行。

必须包含：任务拆分、精确文件路径、关键接口或类型、核心测试用例、验证命令、风险点。关键逻辑和容易误解的边界给代码；普通 glue code 引用现有模式。

### Level 3：执行级计划

适合高风险变更、认证/权限/安全/支付/数据迁移/删除数据、并发/调度/跨平台行为、多 subagent 协作、弱模型或低信任执行。

必须包含：2-5 分钟粒度的 checkbox 步骤、精确文件路径、关键代码或完整代码、精确命令和预期输出、每个任务的验证方式、独立验收清单和提交方式。

## 必须写清

- 用户可见行为、输入输出、业务规则；
- API 合约、数据结构、权限和安全边界；
- 错误处理、兼容性、迁移或回滚要求；
- 修改/创建的文件路径，以及每个文件职责；
- 不做什么；
- 验收标准、验证命令、预期输出、失败判断方式；
- 高风险操作是否可能造成数据丢失、安全风险或线上回归。

## 可以留给执行者

- 内部 helper 如何拆分；
- 局部变量命名；
- 非关键 glue code；
- 与现有代码风格对齐的低风险实现；
- 无业务含义的目录微调。

保留发挥空间的前提是：目标、约束、文件位置和验收标准已经明确。

## 验证要求

每个任务都必须写清：

- 验证的具体行为；
- 使用自动化测试、构建、lint、类型检查、快照、手动操作还是日志检查；
- 精确命令或操作步骤；
- 预期输出；
- 失败时如何判断问题；
- 是否需要另一个 agent 独立验收。

对于业务逻辑、数据转换、权限判断、错误处理、调度和状态机，优先给自动化测试或可重复验证。对于 UI、配置、文档或一次性脚本，可以使用手动验证清单。

## 禁止写法

计划中不要出现无法执行的占位表达：

- `TBD`、`TODO`、`implement later`、`fill in details`；
- “添加适当的错误处理”；
- “添加校验”；
- “处理边界情况”；
- “为上面的内容编写测试”，但没有测试目标、验证方式或测试代码；
- “类似任务 N”；
- 只写做什么，不写关键约束、位置或验收方式；
- 引用前文没有定义过的类型、函数、字段或方法。

## 自查

保存计划前必须自查：

- 规格中的每个需求是否对应到任务或非目标；
- 是否扫描并修复了占位表达；
- 类型、函数、字段、路径命名是否前后一致；
- 当前粒度是否匹配风险；
- 每个任务是否有可执行的验证方式；
- 最终文件是否已经写入计划路径。
