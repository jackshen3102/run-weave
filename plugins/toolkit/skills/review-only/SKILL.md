# 仅评审

## 目的

使用这个技能时，先做高层次的批判性评审，再做代码层面的评审。

核心目标：

- 在局部代码缺陷之前，先发现方向层面的错误
- 质疑当前架构是否真的是合适架构
- 暴露更好的替代方案（自研 vs 购买、简化 vs 定制、平台默认能力 vs 自建能力）

## 触发规则

- 这个技能是手动触发的。
- 仅当用户明确要求 `review-only`（或明确表示“只评审，不要修改代码”）时使用。
- 不要在普通编码任务中自动使用。

## 硬性约束

- 不要编辑任何被评审的源码、配置、测试或产品文档。
- 不要运行任何会修改被评审对象的写操作（`apply_patch`、重定向写入、格式化修复、代码生成）。
- 唯一允许的写操作：评审完成后，将评审结果写入当前仓库的 `docs/review` 目录。
- 不要提交、stash、rebase，或改变 git 状态。
- 不要用局部实现方案代替评审发现。

## 评审方法（自上而下，强制顺序）

1. 问题定义评审（我们解决的是正确的问题吗？）
2. 系统/架构评审（这个形态是正确的吗？）
3. 自研 vs 购买 / 重复造轮子评审（是否重新实现了已有能力？）
4. 可运维性和生命周期评审（它能安全运行、演进和失败吗？）
5. 代码级正确性评审（只能在完成 1-4 之后进行）

不要从逐行代码评论开始。

## 评审范围

1. 代码正确性和回归风险。
2. 架构边界，以及耦合/内聚情况。
3. 测试策略和覆盖盲区。
4. 运维风险（配置、脚本、CI 门禁逻辑、可观测性）。
5. 可维护性（可读性、可扩展性、归属清晰度）。
6. 战略适配度：
   - 需求到设计的可追溯性
   - 总复杂度 vs 业务价值
   - 是否有机会删除组件，而不是改进组件
7. 替代方案：
   - 至少一个更简单的架构
   - 至少一个“使用现有平台/工具链”的方案
   - 权衡对比（成本、风险、交付速度、锁定风险）

## 激进评审启发式

- 在被证明正确之前，先假设当前架构可能是错的。
- 如果结果相同，优先删除层级，而不是打磨层级。
- 如果成熟默认能力可以替代自定义基础设施，要标记出来。
- 标记会增加全局系统复杂度的局部优化。
- 如果运维负担很高，“现在能跑”并不充分。

## 评审强度（自适应）

简单变更使用轻量评审，复杂或高风险变更使用强力评审。

### 轻量模式（简单变更默认）

适用场景：

- 单模块变更，耦合较低
- 文案、样式、小逻辑修复
- 不影响边界、协议、基础设施或部署

预期输出：

- 发现要简洁
- 架构部分可以很短，或明确写出“无架构问题”
- 不强制要求多方案架构对比

### 强力模式（复杂或高风险变更）

满足任一条件时触发：

- 跨模块或跨服务变更
- 协议、数据结构、契约变更
- 基础设施、工具链、门禁变更（构建、测试、部署、质量门禁）
- 安全、鉴权、会话、运行时行为变更
- 有明显运维或成本影响

预期输出：

- 提供至少 2 个实质上不同的架构选项
- 每个选项都包含权衡：交付速度、复杂度、运维风险
- 明确说明：
  - 推荐方案（附理由）
  - 不推荐方案（附理由）

## 强制输出格式

- 先输出发现，并按严重程度排序（`P1` -> `P2` -> `P3`）。
- 每个发现都包含：
  - 简短标题
  - 为什么这是风险
  - 具体文件 + 行号引用
  - 可执行的修复方向（不实际实现）
- 如果没有发现重大问题，要明确说明，并列出剩余风险/测试缺口。

按以下顺序使用两个发现分组：

1. `架构 / 策略发现`（强制保留，只有在明确说明理由时才可以为空）
2. `代码 / 实现发现`

每个架构发现必须包含：

- 当前决策
- 为什么它在系统层面可能是错的
- 更好的候选方案
- 迁移/过渡风险说明

在强力模式下，架构部分必须至少包含 2 个候选方案。

## 结果归档

- 评审结束后，必须创建或更新一份 Markdown 评审报告。
- 报告目录固定为当前仓库下的 `docs/review`；如果目录不存在，先创建。
- 文件名必须以当前日期开头，日期格式为 `YYYY-MM-DD`，例如：`2026-05-04-review.md`。
- 如果能从评审范围推断主题，使用 `YYYY-MM-DD-<主题>.review.md`；主题要简短，并使用小写字母、数字和连字符。
- 如果同名文件已存在，不要覆盖；追加递增后缀，例如：`2026-05-04-review-2.md`。
- 报告内容应与最终回复中的评审结论一致，可增加命令输出摘要、残余风险和检查范围。
- 最终回复里必须给出写入的报告路径。

## 评审行为

- 保持批判性，并基于证据。
- 避免只针对风格的挑刺，除非它们会造成真实风险。
- 报告前，用命令/阅读验证可疑问题。
- 相比外观层面的评论，优先关注架构和行为风险。
- 对“重复造轮子”的情况要明确指出，并给出替代候选。
