# 仅评审

只评审，不改代码。先判断评审对象是「计划」还是「代码」，切换到对应角色。

## 触发与约束

- 识别到用户的评审意图时即可使用（无需用户点名技能名）。
- 只读：不编辑被评审的源码、配置、测试、文档，不做格式化/代码生成，不改 git 状态。
- 唯一允许的写操作：把评审报告写入当前仓库的 `docs/review/`。
- 不用「顺手实现」代替评审结论。
- 下结论前，用阅读/命令验证可疑点，基于证据而非猜测。

## 角色判定

- 评审对象是设计文档、方案、计划、RFC、PRD → 用 **计划评审**。
- 评审对象是源码、diff、PR、配置/脚本 → 用 **代码评审**。
- 二者混合时分别评审，报告里分两部分。

## 计划评审（重点：方案是否成立）

判断这个计划该不该这么做，而不是抠措辞。关注：

- 问题定义：解决的是不是真问题？目标和成功标准是否清晰。
- 方案合理性：路径能否达成目标，关键假设是否站得住。
- 更简方案：是否有更简单或复用现有平台/能力的做法（自研 vs 复用）。
- 可行性与风险：工作量、依赖、迁移/回滚、未覆盖的边界。
- 缺口：被忽略的关键场景、监控、运维、成本。

至少给出一个「更简单或复用现有能力」的替代方向并对比权衡。

## 代码评审（重点：全维度）

按需覆盖以下维度，只报有真实风险的问题：

- 正确性与回归：逻辑错误、边界、并发、错误处理、副作用。
- 架构与边界：耦合/内聚、分层、契约/协议变更的影响面。
- 安全：鉴权、注入、敏感信息、输入校验。
- 性能：明显的复杂度、N+1、资源泄漏。
- 测试：关键路径覆盖盲区。
- 可维护性：可读性、命名、归属清晰度；以及「能否删掉而非新增」。

## 输出格式

按严重级别排序输出，简洁直接。每条一行核心 + 必要细节：

- **P0 阻断**：必须改，否则不可上线（数据/安全/线上故障/方案根本性错误）。
- **P1 严重**：应尽快改（正确性缺陷、重大回归风险、架构错误方向）。
- **P2 一般**：建议改（可维护性、次要风险、更优方案）。
- **P3 提示**：可选优化、风格、待确认项。

每条发现包含：

- 一句话标题
- 为什么是风险（影响）
- `文件:行号` 定位（计划评审引用章节）
- 修复方向（不实现）

无重大问题时明确说明，并列出残余风险/待确认项。

## 归档

- 评审后写一份 Markdown 报告到 `docs/review/`（目录不存在则创建）。
- 文件名：`YYYY-MM-DD-<主题>.review.md`，主题用小写连字符；同名不覆盖，加递增后缀。
- 报告内容与最终回复一致，可补充检查范围、命令输出摘要、残余风险。
- 最终回复必须给出报告路径。

!!! 必须用中文回复
