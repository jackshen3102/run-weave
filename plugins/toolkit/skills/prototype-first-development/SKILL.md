# 原型先行开发

使用这个技能，在产品实现前先把交互意图落成具体可看的东西。产物是一个小型可运行原型，不是生产代码，也不能证明后端、协议或运行时能力已经存在。

这个技能不是“生成一个 HTML 模板”就结束，而是一套从需求到原型、从原型到计划、从计划到实现的开发方式。除非用户明确要求跳过，否则不要压缩或改写这条路径。

## 核心规则

当用户还在收敛行为、布局、状态或工作流时，先创建本地、可读、无需构建的原型。原型稳定后，再更新实施计划；只有用户明确推进实现时，才修改产品代码。

原型的用户可见界面必须按最终产品方案绘制：除技术实现方式和数据来源外，UI、文案、交互入口、状态反馈都应与计划落地后的产品一致。不要在产品画面里加入解释性文案、操作指引、快捷键说明、需求备注、标注、教学卡片或为了让人理解原型而临时增加的按钮。

如果确实需要用于演示的状态切换、mock 触发器或备注说明，必须放在 README，或放在默认隐藏、视觉上与产品画面隔离的 prototype helper 区域，并明确标记为不进入实施计划。任何 agent 都不能从这类辅助内容反推最终产品 UI。

## 默认产物

优先使用这个结构：

```text
docs/prototypes/<feature-slug>/
  README.md
  index.html
  app.js
  mock-state.json
  prototype-preview.png   # 可选，浏览器验证后保存
```

从零开始时，使用技能自带模板 `references/html-prototype-template/`。把它复制到目标仓库并替换占位内容，不要重新发明目录结构。

需要过程检查时，读取 `references/prototype-workflow-checklist.md`。需要从原型整理实施计划时，读取 `references/implementation-plan-handoff-template.md`，并优先结合 `$toolkit:write-plan` 落盘计划。

## 工作流

1. 原型简报：
   - 用户目标
   - 用户动作
   - 主要流程
   - 关键状态
   - 明确的非目标
   - 原型要影响的真实产品界面或模块

2. 可运行原型：
   - 默认保持静态、无需构建，除非仓库已经有更强的约定
   - 首屏只呈现最终产品会出现的界面，不放需求解释、操作教程或设计备注
   - 仅使用最终产品需要的用户操作入口；不要为了演示方便额外加“下一步”“切状态”“模拟事件”等可见控件
   - 把布局和挂载点放在 `index.html`
   - 把状态流转和事件处理放在 `app.js`
   - 把领域对象、模拟事件和样例记录放在 `mock-state.json`
   - 除非用户明确要求代码与原型对齐，否则不要导入产品源码

3. 调整循环：
   - 优先保证可读性，不追求炫技
   - 组件和函数命名贴近业务领域
   - 状态流转保持容易修改
   - 只添加最终产品会保留的交互控件
   - 如需切换 mock 状态，优先用 URL 参数、README 步骤或默认隐藏的 helper 面板，不把它画进产品画面
   - 用户点、改、截图、反馈时，只调整原型，不碰产品代码

4. 原型冻结：
   - 在 README 中更新启动命令、验证点、边界和非目标
   - 浏览器验证重要时，保存 `prototype-preview.png`
   - 在 Runweave 仓库中，浏览器页面复现、修复或验收必须使用 `$toolkit:playwright-cli` 完成打开页面、交互、截图或 DOM 检查
   - 冻结前确保 README 记录了最后采用的交互方案和被放弃的方向

5. 实施计划：
   - 在计划中引用原型目录
   - 区分“原型表达的意图”和“当前代码真实现状”
   - 声称可行前，先检查产品代码
   - 把协议、后端、运行时缺口与 UI 工作分开列出
   - 计划必须写清产品行为、协议变化、代码落点、验收方式

6. 产品实现：
   - 实现时再回到真实代码，不从原型直接推断已有能力
   - 产品代码修改必须以当前代码现状为准
   - 保留原型作为交互基准，但不要照搬原型代码
   - 验证时同时覆盖产品行为和原型中冻结的关键交互

## 质量目标

让生成内容像人类写的、后续能维护：

- 可读：文件名清楚，函数小，状态名朴素，事件处理直观。
- 易调：模拟数据放 JSON，样式按 UI 区域组织，不生成晦涩代码。
- 高效：无构建步骤；除非能明显提升该原型的开发速度，否则不引入框架。
- 真实：用户可见界面不夹带说明、备注或脚手架控件，截图应能直接作为最终产品体验的参考。
- 诚实：README 写清楚哪些是模拟的，真实实现还缺什么。

## 功能分类账

原型中可能出现两类元素，必须显式分开记录：

- 产品核心功能：最终需要落地到产品里的能力、状态、入口、反馈或业务规则。
- 原型辅助功能：为了演示、切换状态、模拟数据、触发事件、方便用户理解或截图而临时加入的控件和面板。

具体分类表和检查项写在原型 README 与 `references/prototype-workflow-checklist.md` 中，`SKILL.md` 只保留原则。

原型辅助功能的默认处理：

- 默认不要出现在用户可见产品画面里；能用 README 或 URL 参数完成的，不要做成 UI。
- 必须与产品画面视觉隔离，并用 `data-prototype-helper="true"` 或等价标识标记。
- 不进入实施计划的产品范围。
- 不作为最终 UI、API、协议或数据模型的依据。
- 如果用户希望保留到产品中，必须在原型冻结时把它改列为产品核心功能或单独写入后续需求。

## 约束

- 不要把原型当成产品合约已经存在的证据。
- 不要把原型代码直接照搬进生产代码。
- 不要模拟每个边界情况；只模拟用户需要看见或验证的关键决策。
- 除非仓库明确要求，否则不要为原型文件新增单元测试。
- 遵守仓库关于浏览器验证、测试层级和文档路径的要求。
- 不要在用户还处于原型调整阶段时提前写实施计划或改产品代码。
- 产品 UI 原型默认使用可运行 HTML；技术架构、事件流或故障因果可视化改用 `$toolkit:architecture-flow-prototyping`。
- 不要把原型辅助功能当作默认产品需求。
- 不要在产品画面里写“快捷键”“操作说明”“这里展示”“用于验证”“模拟状态”等说明性内容，除非这些文案本来就是最终产品 UI 的一部分。

## 启动方式

对于 `docs/prototypes/<feature-slug>` 下的原型：

```bash
python3 -m http.server 6188 --directory docs/prototypes/<feature-slug>
```

打开：

```text
http://127.0.0.1:6188/
```
