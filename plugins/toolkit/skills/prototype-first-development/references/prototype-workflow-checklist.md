# 原型先行开发检查清单

使用这个清单检查一次原型先行流程是否跑完整。不要把它当作普通说明文；它是阶段门禁。

## 1. 原型简报

- 是否写清楚用户目标、用户动作、关键状态和非目标？
- 是否写清楚原型要影响的真实产品界面或模块？
- 是否初步区分了产品核心功能和原型辅助功能？
- 是否避免一开始就讨论生产代码实现细节？

## 2. 可运行原型

- 是否生成到 `docs/prototypes/<feature-slug>/`？
- 是否包含 `README.md`、`index.html`、`app.js`、`mock-state.json`？
- 是否可以通过 `python3 -m http.server` 打开？
- 是否只模拟关键路径，没有连接真实后端？
- 是否把 mock 数据放在 JSON，而不是散落在页面字符串里？
- 每个按钮、面板、状态标签、事件流或模拟开关是否都能归类为产品核心功能或原型辅助功能？

## 3. 调整循环

- 用户反馈时，是否只调整原型，不碰产品代码？
- 每轮调整是否更新 README 的调整记录？
- 是否保持函数、状态、样式命名可读？
- 是否避免引入真实项目组件和复杂依赖？
- 新增的演示控件是否同步写入“原型辅助功能”表？
- 新增的产品能力是否同步写入“产品核心功能”表？

## 4. 原型冻结

- README 是否记录最终采用的交互？
- README 是否记录放弃的方向和原因？
- README 是否确认产品核心功能清单？
- README 是否确认原型辅助功能清单？
- 浏览器验证重要时，是否保存最终截图？
- 在 Runweave 仓库中，是否使用 `$playwright-cli` 完成浏览器页面复现、修复或验收？

## 5. 实施计划

- 计划是否引用原型目录？
- 计划是否区分“原型表达的意图”和“当前代码是否支持”？
- 计划是否只承接产品核心功能？
- 原型辅助功能是否明确列为不进入产品实现？
- 是否先检查真实代码，再写协议、后端、运行时、前端落点？
- 是否写清验收方式和失败判断？

## 6. 产品实现

- 是否只在用户明确推进实现后修改产品代码？
- 是否没有把原型代码直接照搬进生产代码？
- 是否没有把原型辅助功能误实现进产品？
- 是否用真实代码现状校准可行性？
- 是否验证了原型冻结的关键交互？

## 偏差处理

如果发现已经提前改了产品代码、跳过了原型冻结、没有把模板放在技能引用目录下，或没有区分产品核心功能与原型辅助功能，先停下来修正流程产物，再继续后续阶段。
