# 真实复现后修复

坚持一个顺序：**先复现，再解决，再优化**。问题描述、测试结论和 P0/P1/P2 标签都是待验证输入，不是问题已经成立的证据。

## 调用边界

本技能只允许通过 `$toolkit:reproduce-before-fix` 显式调用，不根据普通的“修复问题”“测试失败”或“优化性能”等请求自动触发。

显式调用代表执行复现门槛，不代表承诺修复。处理用户提供的问题，也处理本次任务中发现的新问题；每个问题都独立经过同一门槛。

需要通过新增、录制和分析 AI 诊断日志逐步定位时，显式组合使用 `$toolkit:diagnostic-log-debugging`。该技能负责日志采集与定位方法；本技能仍负责判断真实复现是否成立、是否允许进入修复和优化。

## 什么是真实复现

使用真实产品入口、当前生产代码路径和实际运行中的进程、接口、存储、事件流或传输层，观察真实可见结果。尽量还原报告中的用户角色、操作顺序、数据规模、环境和时序。

可以用脚本或浏览器自动化驱动真实系统，但自动化只能代替人工操作，不能代替被验证的系统。以下内容可以帮助形成假设，但不能单独算作复现：

- mock、stub、fake transport 或伪造响应；
- 只构造 fixture、内存对象或虚假业务数据；
- 绕过产品入口直接调用内部类、私有函数或测试专用接口；
- 只阅读代码后推断“应该会发生”；
- 只让单元测试或模拟环境失败。

## 执行流程

### 1. 把结论还原成可观察问题

先写清触发条件、操作入口、预期结果和实际症状。区分报告原话、当前事实和待验证推测。优先级只影响处理顺序和响应速度，不得跳过复现。

如果描述缺少细节，先从代码、测试记录、日志和实际产品入口补齐；只有缺失信息会实质改变复现路径时才向用户确认。

### 2. 在修改前执行真实复现

记录代码基线、运行环境、真实入口、完整步骤和观察证据。证据按问题选择，可以是页面行为、DOM、网络消息、真实接口响应、日志、存储结果、事件序列、截图、录屏或性能数字。

在安全且成本合理时重复执行，排除偶发环境噪声。性能问题必须先记录基线、规模和可判断的数字；没有测到问题，就不进入性能优化。

### 3. 通过门槛决定动作

只允许以下结论：

| 结论         | 判断依据                                                 | 动作                                   |
| ------------ | -------------------------------------------------------- | -------------------------------------- |
| 已复现       | 真实入口稳定出现报告症状，并有可回查证据                 | 定位已证实机制，进入最小修复           |
| 未复现       | 已覆盖可信路径，但真实系统没有出现症状                   | 停止，不修改代码，记录尝试与证据       |
| 边界情形     | 超出支持契约，或只能靠伪造内部状态、不可达时序等方式触发 | 停止，不为该情形扩展实现               |
| 环境阻塞     | 缺少设备、权限、服务、数据或必要版本，无法完成真实路径   | 停止修改，报告阻塞；不要写成“未复现”   |
| 实际问题不同 | 真实复现暴露了不同症状、范围或根因                       | 以证据重述问题；范围实质变化时先行对齐 |

低频不自动等于边界情形。只要受支持的真实用户可以从正常入口触发，即使概率低，也属于有效问题。反过来，一旦契约和真实入口已经证明某情形不可达，就不要继续扩大输入空间来制造复现。

### 4. 只修复已证实的机制

改动必须直接对应复现证据和已定位机制，不顺手处理相邻推测、未复现风险或其它优先级问题。若同时拿到 P0、P1、P2，分别建立结论；若只拿到 P2，也按同一流程处理，不补造更高优先级任务。

### 5. 用同一真实场景验证修复

先原样重跑修复前的复现场景，确认实际症状消失且预期结果成立，再执行受影响路径的回归。静态检查、代码阅读或 mock 测试不能替代这一轮验证。

只有正确性问题已经解决且真实测量仍显示可优化空间时，才进入优化。优化后继续使用同一基线和规模比较，不用主观描述代替数字。

## 交付结果

按问题给出简洁、可审计的结果：

- 问题与原始优先级；
- 真实环境和复现步骤；
- 预期、实际与证据；
- 门槛结论；
- 是否修改代码及修改范围；
- 修复后同场景结果；
- 未处理原因或环境阻塞。

不要把“尝试修复”“理论上修复”或“测试应该通过”写成完成。没有真实复现，就没有修复。
