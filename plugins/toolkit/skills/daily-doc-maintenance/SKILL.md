# 每日文档整理

这个 skill 用于每天整理 Runweave 文档，让文档反映当前功能和稳定边界。它只更新或删除文档，不修改代码。

“每天定时”必须由外部调度器触发，例如 GitHub Actions、launchd/cron 或团队 agent 平台。skill 本身只定义 Codex 被触发后的执行流程。

## 硬边界

- 只允许改 `docs/**/*.md`、`README.md` 中的文档入口、文档引用的图片/草图资产，以及本 skill 自身被明确要求更新时的 skill 文档。
- 禁止修改 `src/`、`server/`、`electron/`、`packages/`、`scripts/`、测试、配置、lockfile、构建产物。
- 不新增前端单测、Vitest、coverage 配置；文档整理不触发前端 TDD。
- 不因为文档发现代码问题就顺手修代码；只在最终报告里列出“代码侧待处理”。
- 不创建长期新文档，除非用户明确要求。默认把新增或零散文档合并进已有权威文档，然后删除冗余文档。

## 文档原则

文档写当前事实和核心判断，不写实现游记。

保留：

- 当前功能是什么、用户怎么感知、什么时候使用。
- 稳定架构边界、关键数据/协议契约、模块职责。
- 重要限制、风险、运维/验证入口。
- 必须长期保留的命令、环境变量、路由、协议字段。

压缩或删除：

- 文件逐行说明、函数调用流水账、临时调试过程。
- 已完成计划中的中间步骤、commit 过程、个人执行记录。
- 与当前代码不一致的旧路径、旧命名、旧截图说明。
- 可从代码直接读出的低价值细节，例如私有函数名、大段代码片段、局部变量解释。

写法要求：

- 中文优先，短段落，标题直接。
- 先写结论和边界，再写必要细节。
- 一个文档只承担一个主题；不要把设计、排障、部署、测试入口混在一起。
- 需要图时只保留能帮助理解边界或流程的图，不把临时验证截图当长期文档资产。

## 范围

- 默认整理范围：`main` 分支从上一天 00:00 到当前的所有 commit。
- 计算时间窗口前，必须先 `fetch` 并让本地 `main` fast-forward 到最新 `origin/main`；如果无法 fast-forward，先停止并说明分叉状态。
- 范围内的文档变更必须逐一检查：新增文档是否需要沉淀进权威文档，旧文档是否过时，文档入口是否需要调整。
- 范围内的代码变更只用于判断文档是否需要更新；即使发现代码问题，本 skill 也只记录待处理项，不修改代码。
- 如果调用方显式提供 Git range，以调用方 range 为准，但仍必须先确认目标 base branch 与远端状态一致。

## 默认输入

优先使用调用方给出的 Git range：

- `base_branch`，默认 `main`。
- `before_ref` 和 `after_ref`，或等价的 `before_ref..after_ref`。
- 本次是否允许删除文档，默认允许删除已合并且确认冗余的文档。

如果没有明确 range：

- 人工触发时，使用当前工作区 diff 和最近 24 小时提交作为近似范围，并在报告里说明这是近似。
- 自动调度时，如果没有 range 或上次游标，不推进长期状态；只做一次性整理报告或本地文档 patch。

## 工作流程

1. 检查工作区。
   - 运行 `git status --short`。
   - 如果存在非文档改动，不覆盖、不回滚；可以读取代码判断事实，但本轮只改文档。
   - 如果存在用户未提交的文档改动，先读清楚再继续，避免覆盖用户正在写的内容。

2. 收集变更。
   - 查看本次 range 的 `git log --oneline` 和 `git diff --name-status`。
   - 分成三类：新增/修改文档、删除文档、代码或配置变化。
   - 对代码变化只做“是否影响文档”的判断，不改代码。

3. 建立文档地图。
   - 先读 `docs/README.md`，确认权威入口。
   - 按主题映射到现有文档：架构、终端、Preview、AI 协作、质量、测试、部署。
   - `docs/superpowers/plans/` 默认视为计划/过程材料，不作为当前功能权威来源；其中仍有效的结论应沉淀到 `docs/architecture/`、`docs/quality/`、`docs/testing/` 或 `docs/deployment/`。

4. 判断是否需要更新。
   - 用户可见功能、协议、路由、状态模型、验证命令、部署方式发生变化：更新权威文档。
   - 只是内部重构且外部行为、边界、命令都没变：通常不更新文档，只在报告里说明。
   - 新文档与已有文档重复：把核心内容合并到权威文档，删除新文档或旧冗余文档。
   - 文档与当前代码冲突：以当前代码为准改文档；无法确认时标注为待确认，不编造。

5. 执行文档整理。
   - 优先更新已有权威文档，不新建长期文档。
   - 更新 `docs/README.md` 的路由表，确保入口能找到当前权威文档。
   - 删除文档前必须确认：核心内容已合并、没有被 `docs/README.md` 或其他文档继续引用、不是仍有价值的历史决策记录。
   - 删除图片/草图前必须确认没有文档引用。

6. 控制细节密度。
   - 把“实现怎么写”改成“系统现在承诺什么”。
   - 细节只保留到能支持维护者判断边界、排障或验证。
   - 如果一个段落主要在解释某个私有函数如何串联，改写为模块职责或删除。

7. 验证。
   - 运行 `git diff --check`。
   - 运行 `git diff --name-only`，确认只有允许的文档范围变化。
   - 删除文档或资产后，用 `rg` 检查仓库内没有残留链接或过期引用。
   - 不为纯文档整理运行前端单测；只有用户要求或文档工具链需要时才补充更重验证。

8. 提交、push 和创建 PR。
   - 验证通过后必须提交、push，并创建 PR。
   - 提交前再次确认 diff 只包含允许的文档范围变化。
   - 分支名使用 `codex/daily-doc-maintenance-YYYYMMDD-HHMM`；如已存在，追加短后缀避免覆盖。
   - commit message 使用 `docs: refresh daily documentation`，或按实际整理主题使用更具体的 `docs: ...`。
   - PR target 使用本次范围对应的 `base_branch`。
   - PR 描述必须包含：整理范围、更新的权威文档、删除的冗余文档、未改原因、验证命令和结果。

9. 输出结果。
   - 列出更新了哪些权威文档、删除了哪些冗余文档。
   - 说明哪些代码变化不需要文档更新，以及原因。
   - 列出 commit、push 分支和 PR 链接。
   - 如果发现代码和文档事实无法对齐，列出待确认问题，不修改代码。

## 删除规则

可以删除：

- 已被权威文档吸收的重复文档。
- 只记录一次性执行过程、临时方案或已过期计划的文档。
- 指向旧架构、旧路径、旧命令且没有长期参考价值的文档。
- 已经没有引用的文档图片或草图资产。

不要删除：

- `docs/README.md` 路由中仍作为入口的权威文档。
- 仍在解释当前功能边界、协议或验证路径的文档。
- 明确作为历史决策、复盘或设计依据保留的文档，除非用户要求清理。
- 不确定是否仍被外部引用的公开文档；这种情况先在报告里标记。

## 最终报告格式

回复保持简短，包含：

- `更新`：改了哪些文档，核心变化是什么。
- `删除`：删了哪些文档/资产，为什么可以删。
- `未改`：哪些变更判断后不需要文档更新。
- `验证`：实际运行的检查命令和结果。
- `提交`：commit、push 分支和 PR 链接。
- `待确认`：需要人工判断的外部事实或产品取舍。
