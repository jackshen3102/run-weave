# Code-Grounded Requirements

把需求落到**真实代码**上再下结论，对抗「不读代码就脑补方案」。

要对齐三件事：当前代码已经怎么做了 / 用户想改成什么 / 两者差在哪。

## 触发边界

- **触发**：改动现有行为、排查「为什么现在是这样」、需求涉及已有组件/状态/接口/链路。
- **不触发**：全新文件从零搭建、纯问答、用户已给出明确实现指令。

## 硬规则

- **先读代码，再梳理需求**，不允许只凭 query 或通用经验下结论。
- 优先读与需求最相关的真实代码：页面/组件、路由、hooks/store、服务层/API、后端接口，以及（若涉及）配置/Electron bridge/storage。
- 排查「为什么现在这样」时，必须看完整链路：调用方 → 中间层 → 真正生效层。
- 关键判断：能力是**已存在但链路没接上（接线/回归问题）**，还是**真的缺能力**——别把前者误判为后者。
- 需求对齐前不进入实现。建议可以给，但必须基于代码现状。

## 证据标准

- 结论要带关键文件路径/模块名，且能被代码事实支撑。
- 推测必须标注：「基于当前代码，我推测……」，不能把推测写成事实。

## 输出格式

按下面顺序，按需裁剪：

### 当前现状

基于代码说明系统现在怎么工作，给关键文件路径。

### 需求理解

用更精确的话重述目标，点明这次改的是行为 / 存储 / UI / 路由还是接口。

### 差异与影响

现状和目标差在哪（行为、数据模型、存储语义、权限、路由跳转），以及可能受影响的模块。

### 建议

1 个推荐方向（必要时补 1 个备选），每条说明为什么贴合当前代码——是最小修复还是结构性改造、是否复用已有接口/组件。

### 待确认点

只在确实需要时保留，1-3 个高价值边界问题；能从代码看出来的不要反问。

默认用中文回复。
