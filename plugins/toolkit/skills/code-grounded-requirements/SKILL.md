# Code-Grounded Requirements

## Purpose

显式调用这个技能时，先结合当前项目代码梳理需求，再给出结论。

目标不是复述用户的话，而是把下面三件事对齐：

- 当前代码已经怎么做了
- 用户现在想改成什么
- 两者之间到底差在哪里

## Hard Rules

- **必须先读代码，再梳理需求。**
- 不允许只根据用户 query 或通用经验直接输出需求理解。
- 必须优先读与该需求最相关的真实代码：
  - 页面/组件
  - 路由
  - hooks / state
  - 服务层 / API 调用
  - 后端接口
  - 配置 / Electron bridge / storage（如果需求涉及这些）
- 如果代码里已经存在能力，只是链路没接上，要明确指出这是“接线问题/回归问题”，不要误判为“缺少能力”。
- 建议可以给，但**必须基于代码现状**，不能给脱离当前实现的空泛建议。
- 在需求尚未对齐之前，不要直接进入实现。

## Workflow

### 1. 锁定相关代码范围

先根据用户 query 判断最可能相关的代码区域。

优先定位：

- 入口页面和路由
- 触发该行为的组件
- 共享 hook / store
- 请求服务或后端接口
- 与存储相关的 localStorage / session / Electron preload

如果需求涉及“现在为什么这样”，必须同时看：

- 调用方
- 中间传递层
- 真正生效层

### 2. 提取代码事实

只记录和当前需求有关的事实，例如：

- 哪个组件负责展示
- 哪个回调负责切换
- 当前 token / connection / settings 是全局还是按 scope 存
- 当前路由在什么条件下跳转
- 当前后端有没有现成接口
- 当前 UI 是静态展示还是可交互控件

不要泛泛总结整个模块。

### 3. 对齐现状和目标

输出时必须明确区分：

- 当前现状
- 用户目标
- 中间差异

差异最好具体到：

- 行为差异
- 数据模型差异
- 存储语义差异
- 权限/可编辑性差异
- 路由跳转差异

### 4. 提出代码约束下的建议

你可以给建议，但要基于当前代码判断，例如：

- 这是最小修复还是结构性改造
- 当前组件是否适合抽成复用组件
- 当前后端是否已有接口，还是应该新增专用接口
- 当前状态模型能不能承载需求，还是需要按连接/按作用域重构

建议必须说明**为什么贴合当前代码**。

### 5. 识别待确认点

如果还有不明确的地方，只提 1-3 个最关键的边界问题。

问题应该是高价值的，例如：

- 这个状态要不要按连接隔离？
- 这个默认项是否允许编辑/删除？
- 校验有效性是本地判断还是服务端判断？

不要把明显能从代码里看出来的事情再反问用户。

## Output Format

默认使用下面的结构，按需裁剪，但顺序不要乱：

### 当前现状

- 基于代码说明现在系统是如何工作的
- 必要时给出关键文件路径

### 需求理解

- 用更精确的话重述用户想要的目标
- 明确指出这次要改的是行为、存储、UI、路由还是接口

### 差异与影响

- 说明现状和目标之间差在哪
- 标出可能受影响的模块

### 建议

- 给出 1 个推荐方向
- 如果有必要，可补 1 个备选方向
- 每条建议都要基于当前代码说明原因

### 待确认点

- 只有在确实需要时才保留
- 控制在 1-3 个

## Evidence Standard

输出结论时，至少要满足以下标准：

- 提到关键文件路径或关键模块名
- 判断必须能被代码事实支撑
- 不能把“推测”写成“事实”

如果是推测，要明确写：

- “基于当前代码，我推测……”
- “从现有调用链看，更像是……”

## Constraints

- 不要一上来输出实现方案细节。
- 不要在没有读代码的情况下下结论。
- 不要把需求梳理写成一份大而空的架构文档。
- 默认用中文回复。
