# 编写测试用例

核心原则：

> **测试用例是「可执行、可取证、可追溯到需求」的验收合约，不是happy-path清单，也不是留给人猜的模糊描述。**

这个 skill 只负责**写**用例。执行用例、修问题由 `run-test-cases` 负责，不要在这里跑测试或改实现。

## 使用边界

适用：已有明确需求/改动，需要产出或补全测试用例文档；写完实施计划要配套验收用例；评审发现用例缺失或不标准。

不适用：

- 需求还没定清、连「做成什么算对」都判断不了 → 先补 spec 或用 `code-grounded-requirements` / `write-plan`；
- 用户要求的是**执行**已有用例 → 用 `run-test-cases`；
- 纯脑暴、无需落地验收合约。

## 验证方式硬约束

- **禁止用静态检查冒充验证。** `typecheck` / `lint` 只是前置门禁，不能作为功能/行为/UI 用例的通过证据。
- **默认用真实环境取证。** 浏览器页面路径的用例验证方式必须落到 `$playwright-cli` 操作真实浏览器；桌面端联动落到 `$computer-use`；后端/协议落到可执行脚本 + 关键输出。不要写「本地 npm run dev 后肉眼看」这类不可取证的步骤。
- **不假设仓库能跑测试框架。** 部分仓库不写、也不允许新增任何测试代码（如无 `*.spec.ts`、无 E2E 测试套件）。动手前探测目标仓库的测试约定，遵循它；不确定时默认「不新增测试代码文件，用真实浏览器/真实环境行为核对」。
- 每条用例都要能追溯到本次需求来源；不给相邻无关功能顺手补用例。

## 落盘位置与格式

- **先探测当前仓库的约定，不要假设。** 动手前查一次：是否已有测试用例目录（常见 `docs/testing/`、`docs/test-cases/`、`tests/cases/` 等）和现成用例文档。
  - 已有同类用例：沿用其目录、命名和写法，保持一致；只**追加**本次需求相关的用例，不要为了统一风格重写既有大文档。
  - 没有任何参照：用 `assets/test-cases-template.md` 起一份新文档，落到 `docs/testing/<主题>-test-cases.md`（目录不存在先创建）。
- 主题用小写连字符（例：`login-flow-test-cases.md`）。**关键是包含以下信息，而不是套某个固定骨架**——范围（测什么/不测什么）、前提事实（代码/接口/环境依赖）、必跑命令、每条用例（ID + 四要素）、验收通过标准。既有文档已用别的结构承载了这些信息，就顺着它写，不要改结构。
- 用例 ID 用大写缩写前缀 + 三位序号（如 `LOGIN-001`），一份文档内唯一、连续；沿用既有文档时跟随其现有编号规则。

## 编写流程

1. **先落地到真实事实。** 读相关实现、协议、现有用例，写「前提事实」小节：入口、关键路径、接口/数据结构、环境变量、依赖。用例必须踩在真实代码路径上，不能是假设。
2. **用系统化方法推导用例，别拍脑袋。** 对每个输入/行为，套用行业成熟的黑盒设计技术生成用例，避免只写 happy path 或凭直觉漏场景。技术选型与示例见 `references/design-techniques.md`：等价类划分、边界值分析、判定表、状态迁移、场景/用例法、错误猜测。
3. **对着覆盖清单查漏。** 逐项过 `references/coverage-checklist.md`：正常路径、边界值、错误态、异步/迟到参数、并发、权限鉴权、幂等/去重、回归点。不适用的维度显式写「不覆盖 + 原因」，不要静默漏掉。
4. **每条用例写全四要素**：前置条件（Given）→ 操作（When）→ **预期结果（Then，可取证）** → **失败判断**。
   - 每条用例围绕**一个**明确行为，标题描述「什么条件下期望什么行为」，即使实现被重写也依然成立——不要用「XX 功能正常」这类无意义标题。
   - 预期与失败判断具体到能对照文件/接口/UI 状态，禁止「功能正常」这类空话。
   - 用例之间相互独立、自带前置，不依赖前一条留下的状态；断言确定、可复现，不依赖时间/顺序/网络抖动等 flaky 因素。
   - 验证**可观察行为**而非内部实现细节（不断言私有函数被调用几次这类实现耦合）。
5. **标注验证方式**：自动化脚本 / `$playwright-cli` 真实浏览器 / `$computer-use` 桌面 / 手动核对。业务逻辑、状态机、协议优先自动化脚本；UI、配置、一次性脚本可手动清单。
6. **写「必跑命令」**：按顺序列出 typecheck、lint、相关 verify/自动化脚本、`git diff --check`（按目标仓库实际有的命令裁剪），标明「任一失败即停」。
7. **写验收通过标准**：把所有用例收敛成一组必须同时满足的条件。

## 落盘前自查

- 每个需求点都对应到用例或显式非目标；覆盖清单每个维度都已判断（覆盖 / 不覆盖+原因）；
- 每条用例只测一个行为、标题描述行为而非函数名、相互独立、断言确定不 flaky、测行为不测实现细节；
- 没有不可执行的占位（TBD、「加适当校验」「测一下边界」这类模糊表达）；
- 每条用例都有可取证的预期结果和失败判断，浏览器路径落到 `$playwright-cli`；
- 引用的文件路径、接口、环境变量、用例 ID 前后一致且真实存在；
- 文件已写入用例路径，最终回复给出路径。

## 与其他 skill 的衔接

- 由 `write-plan` 触发时：计划的验收章节应引用本文件产出的用例路径，做到「计划完成 = 用例齐备」。
- 用例写完后要执行：交给 `run-test-cases`，不要在本 skill 内跑。
- 用例设计技术：见 `references/design-techniques.md`；覆盖维度：见 `references/coverage-checklist.md`。
