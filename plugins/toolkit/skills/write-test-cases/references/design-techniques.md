# 测试用例设计技术

系统化推导用例的黑盒方法（对齐 ISTQB / 业界通用实践）。目的：用最少的用例覆盖最多的真实风险，避免只写 happy path 或凭直觉漏场景。

对每个输入/行为，按下表选一到多种技术组合使用，不要只用一种。

## 目录

- 何时用哪种
- 等价类划分
- 边界值分析
- 判定表
- 状态迁移
- 场景/用例法
- 错误猜测
- 好用例的质量属性
- 要避开的测试坏味道

## 何时用哪种

| 输入/行为特征                        | 首选技术                |
| ------------------------------------ | ----------------------- |
| 输入有取值域、可分组                 | 等价类划分 + 边界值分析 |
| 多个条件组合决定不同结果             | 判定表                  |
| 行为依赖当前状态、有状态跳转         | 状态迁移                |
| 端到端业务流程、多步操作             | 场景/用例法             |
| 有经验判断易错点（空值、并发、超时） | 错误猜测                |

## 等价类划分（Equivalence Partitioning）

把输入域切成「系统按同样方式处理」的子集，每个子集取一个代表值即可，不必穷举。

- 划**有效等价类**（符合规范）和**无效等价类**（违反规范），两者都要覆盖。
- 每个无效等价类单独一条用例，避免一条里塞多个非法输入互相掩盖。
- 示例：用户名长度 6–12 → 有效类「6–12 位」；无效类「<6 位」「>12 位」「含非法字符」「空」。

## 边界值分析（Boundary Value Analysis）

Bug 集中在边界。在每个等价类的边界及其邻域取值。

- 取：min、min−1、min、max、max+1（以及 0、空、溢出等特殊边界）。
- 示例：年龄 18–60 → 测 17 / 18 / 19 / 59 / 60 / 61。
- 对分页、长度、配额、时间窗口等数值/容量参数尤其必要。

## 判定表（Decision Table）

多个条件组合出不同结果时，列全条件组合，避免漏掉某种组合。

- 行是条件与动作，列是规则（每种组合一列）。
- 覆盖每一条规则至少一条用例；对不可能组合标注并排除。
- 示例：登录（账号存在? / 密码正确? / 账号锁定?）→ 组合出「成功 / 密码错 / 用户不存在 / 已锁定」等结果。

## 状态迁移（State Transition）

行为依赖当前状态时，画状态机，覆盖状态与迁移。

- 覆盖：每个合法迁移各一条；关键的**非法迁移**（在某状态触发不该允许的操作）各一条。
- 关注初始态、终止态、以及重启/重连后的状态恢复。
- 示例：任务 idle→running→completed；测非法迁移如 completed 再次 start 应被拒。

## 场景/用例法（Scenario / Use Case）

端到端业务流程用「基本流 + 备选流 + 异常流」组织，贴近真实用户路径。

- 用 **Given–When–Then** 描述每条：Given 前置上下文 / When 触发动作 / Then 可观察结果。
- 基本流（happy path）一条；每个备选流、每个异常流各一条。
- 场景之间保持独立，各自 Given 自带前置，不依赖上一条的遗留状态。

## 错误猜测（Error Guessing）

基于经验补充易错点，作为上述技术的补漏，不作为主力：空/null、超长、特殊字符、并发冲突、超时、重复提交、网络中断、权限越界。

## 好用例的质量属性

每条用例应满足：

- **单一行为**：只验证一个行为，一个明确的预期。
- **描述性标题**：写「在什么条件下，期望什么行为」，实现被重写后标题依然成立；不要用函数名或「XX works」。
- **独立**：自带前置，不依赖其他用例的执行顺序或遗留状态，可单独/并行跑。
- **确定**：断言明确、可复现，不依赖时间、随机、顺序、网络抖动（否则是 flaky）。
- **测行为不测实现**：断言可观察的输出/状态，不断言私有函数被调用几次、内部字段等实现细节。
- **可取证**：预期能对照真实文件/接口响应/UI 状态；失败判断写清「出现什么现象即判失败」。

## 要避开的测试坏味道

- **测实现细节**：断言内部调用次数、私有方法、内部字段 → 实现一重构就误报。
- **永不失败的用例**：预期写得太宽（如「不抛异常就算过」）→ 无法暴露问题。
- **flaky 用例**：依赖时间/顺序/网络/随机种子，或断言浮点相等、断言返回顺序。
- **无意义标题**：标题只是函数名或「功能正常」→ 读者无法判断测的是什么行为。
- **相互依赖**：一条用例依赖另一条的副作用 → 一处失败级联误报，且不能并行。
- **一条塞多个断言/多个非法输入**：失败时定位不到根因，且互相掩盖。
