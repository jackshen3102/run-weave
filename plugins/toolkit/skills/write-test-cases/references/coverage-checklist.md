# 测试用例覆盖清单

编写用例时逐项过一遍。每个维度要么给出对应用例，要么显式写「不覆盖 + 原因」，不要静默漏掉。

## 目录

- 功能正确性
- 边界与异常
- 状态与时序
- 数据与协议
- 安全与权限
- 回归与兼容
- Case 原子性与执行边界
- 可取证性（横切要求）

## 功能正确性

- 主路径（happy path）：需求描述的核心行为按预期发生。
- 需求里每条业务规则至少一条用例。
- 多入口触发同一功能时，每个入口分别覆盖（UI 按钮 / CLI / API / 快捷键）。

## 边界与异常

- 空值 / 缺参 / 超长 / 非法输入 / 极值。
- 依赖不可用：后端 down、网络失败、文件缺失、目录不存在、外部进程未安装。
- 错误态的用户可见反馈是否正确（报错文案、降级、fallback），以及降级后主流程是否仍成立。
- 区分「预期内 fallback」（应 warning）与「真失败」（应 error）——本仓库约定别混用。

## 状态与时序

- 异步/迟到参数：晚到的 async 结果不能破坏已进入的状态。
- 并发：多 tab / 多 panel / 多 thread / 多后端同时操作时的隔离，不串扰。
- 状态机迁移：running / idle / completed 等状态的每条合法迁移与非法迁移。
- 重启 / 重连恢复：进程重启后状态能从持久化恢复，消费者能重连。
- 幂等与去重：相同 dedupeKey / 重复事件不产生重复副作用。

## 数据与协议

- 持久化文件（event log、projection、lock、token）的真实内容可取证。
- 前后端 / CLI / Electron 共享协议变更的影响面都被覆盖。
- 敏感字段（token、Authorization）不被写入日志 / 同步镜像 / manifest。

## 安全与权限

- 鉴权：受保护接口缺 token / 错 token 被拒。
- Origin、query 参数、payload 校验。
- 越权访问隔离（不同 project / agent / thread 不可见彼此数据）。

## 回归与兼容

- 本次改动明确列出的「不能回退的旧行为」都有守护用例。
- 向后兼容路径（旧格式 / 旧字段 / legacy lock / backend fallback）仍可用。
- 已知历史 bug 的复现用例（防止再次回归）。

## Case 原子性与执行边界

对每条候选 case 做一次生成前审计：

- 主不变量是否只有一个；失败时能否直接指出哪条行为合同未满足。
- 前置条件能否由本 case 自己建立，不依赖前一条 case 的副作用。
- fixture、surface、endpoint、target、WebContents 和 UI ref 的归属与生命周期是否覆盖全部步骤。
- 是否混入两个可独立失败或恢复的能力轴；如果是，拆成独立 case。
- 是否可能出现一半已有 pass 证据、另一半因不同环境能力缺失而 blocked；如果是，拆分以保留 verdict。
- 中途重启或重建资源时，是否只有该生命周期变化是主不变量，并由同一执行者重新取得 fresh 引用、
  完成 after 取证与 cleanup；否则拆分。
- fail-fast 停在任一步时，是否会永久跳过另一个 required 不变量；如果会，拆分。

同一不变量下的等价输入、边界值和一次状态迁移的 before/after 不需要机械拆分。

## 可取证性（横切要求，每条用例都要满足）

- 预期结果具体到可对照的文件内容 / 接口响应 / UI 状态，不是「功能正常」。
- 失败判断写清「出现什么现象即判失败」。
- 浏览器路径用 `$toolkit:playwright-cli` 取真实证据；桌面联动用 `$computer-use`；后端/协议用 verify 脚本或临时 Node 脚本。
- 禁止用静态检查（typecheck/lint）或代码阅读冒充 UI/行为验证。
