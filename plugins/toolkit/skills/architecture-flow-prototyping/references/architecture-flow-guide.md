# HTML 架构流程设计指南

本指南提供可选构图方法，不是页面模板。先确定核心目标，再挑需要的部分；不要为了满足指南而增加视图或交互。

## 目录

1. 选图规则
2. 页面层级
3. 场景设计
4. 问题因果
5. 数据模型
6. 视觉与交互

## 1. 选图规则

根据要解释的关系选择最小视图：

| 要解释的关系                        | 推荐视图 |
| ----------------------------------- | -------- |
| 跨模块、跨进程、跨 ownership 的流动 | 分层泳道 |
| 调用先后、同步异步、竞态            | 时序图   |
| 生命周期、状态迁移                  | 状态机   |
| 接口、事件、字段映射                | 表格     |
| 一个触发影响多个下游                | 因果链   |

多数情况下不需要用一张图承担所有关系。圆形图更适合少量对等节点；端到端因果和恢复语义通常有其它更清楚的表达，但仍应由具体材料决定。

## 2. 页面层级

复杂诊断型原型可以采用：

```text
标题与范围
关键数字
视图导航
场景选择
分层主链 ┃ 当前场景步骤与指标
```

按需加入下钻视图：

- 问题因果：问题列表 + 触发/机制/结果/证据
- 事件时序：参与者、消息方向、同步或异步边界
- 接口与事件：协议筛选 + 契约表

纯流程型原型可以像 `agent-team-loop-flow` 一样使用单页长图，依次展示生命周期、核心 loop、出口和关键参数，不必加入导航或交互。首屏不要平铺全部文件、类、接口和问题。

## 3. 场景设计

只有当不同运行条件会显著改变路径或帮助暴露问题时才设计场景。可选场景包括：

- 并发写入
- 断线与积压恢复
- 重启和保留期
- 多 backend / 多实例 ownership
- 同一语义的多入口
- 写入关键路径与规模增长

如果加入场景，通常包含：

- 一句话结果
- 真实经过的节点 id
- 2–4 个关键数字
- 3–6 个有顺序的步骤
- 关联问题 id

## 4. 问题因果

诊断型原型优先使用：

```text
触发条件 → 代码机制 → 可见/数据结果 → 证据
```

纯流程说明不必加入问题编号或证据卡片。不要把建议放进问题事实；目标架构和修复选择必须单列，并只在用户要求时出现。

证据优先级：

1. 真实运行复现、日志、DOM、网络消息或性能数字
2. 当前代码的确定控制流
3. 协议缺少某种表达能力
4. 基于结构的风险推测

只有前两级通常足以进入修复；协议缺口需要可见失败或用户明确决策，风险推测只保留观察。这是诊断建议，不限制用户明确选择的架构设计任务。

## 5. 数据模型

事实较多或需要交互时，把它们放进 `mock-state.json`：

```json
{
  "nodes": [
    {
      "id": "event-store",
      "lane": "app-server",
      "title": "EventStore.append",
      "responsibility": "分配 id 并追加 JSONL",
      "source": "app-server/src/event-store.ts"
    }
  ],
  "scenarios": [
    {
      "id": "backlog",
      "label": "积压恢复",
      "path": ["event-store", "catchup", "consumer"],
      "metrics": [
        { "label": "待恢复", "value": "120" },
        { "label": "已交付", "value": "100" }
      ]
    }
  ],
  "issues": [
    {
      "id": "P1",
      "status": "已复现",
      "trigger": "cursor 后积压超过 100 条",
      "mechanism": "只发送一批 catchup",
      "consequence": "历史事件未交付",
      "evidence": "120 条只收到 100 条"
    }
  ]
}
```

简单静态流程可以直接写进 HTML。只记录与核心问题有关的事实；主键、ownership、接口方向、事实源、projection、cursor、保留期和批大小按需选择。

## 6. 视觉与交互

- 可以使用稳定颜色表示角色，但不要用颜色代替文字状态。
- 非当前场景节点可以弱化，但不能完全消失。
- 节点点击只显示职责边界和证据，不改变架构事实。
- 如果存在场景切换，同步更新路径、数字和步骤。
- 如果存在问题状态，使用文字，不只用 P0/P1 和颜色。
- 接口数量较多时可按 REST、WebSocket、internal 等协议筛选。
- 桌面技术图优先保证 1440–1600px 可读，再处理小屏滚动。
