# HTML 架构流程原型自检提示

按任务选择有价值的问题，不要求逐条满足。核心判断始终是：原型是否讲清了目标关系，并帮助发现值得验证的潜在问题。

## 1. 范围与事实

- 核心目标是否明确？
- 是否记录主要读者、代码基线、事实源和日期？
- 是否明确非目标和 ownership 边界？
- 是否读取调用方、中间层和真正生效层？
- 关键关系是否能回到代码、接口、日志或实验？

## 2. 信息结构

- 首屏是否直接服务于核心问题？
- 是否选择了合适的泳道、时序、状态机、表格或因果视图？
- 选择的表现形式是否真的帮助理解，而不是只增加节点和连线？
- 需要同时出现对象、接口、时序或问题时，是否形成清晰层级？
- 存储、projection、cache 和 cursor 如果与主问题相关，是否与主数据流区分？

## 3. 场景与问题

- 如果存在多个场景，是否都与用户问题直接相关？
- 如果加入场景，路径、步骤和关键数字是否同步？
- 如果承担问题诊断，每个问题是否包含触发、机制、结果、证据和来源？
- 如果承担问题诊断，是否区分已复现、未复现、明确边界和性能未达门槛？
- 是否避免在复现前提出重构或目标架构？

## 4. 原型产物

- 是否生成到 `docs/architecture-flows/<architecture-slug>/`，而不是 `docs/prototypes/`？
- 是否至少包含 README 和可运行 index.html？
- 是否能通过本地 HTTP server 打开？
- 复杂数据是否与页面布局合理分离？
- 页面交互是否只服务于架构阅读和诊断？
- 是否区分架构说明核心与 prototype helper？

## 5. 修复后同步

- 是否更新主链和节点职责？
- 如果相关，是否更新场景数字、问题证据、接口或事件契约？
- 是否更新 README 与默认截图？

## 6. 浏览器冻结

- 是否实际执行 `$playwright-cli`？
- 默认视图是否能看到主链？
- 原型实际提供的交互是否都可用？
- 原型实际展示的状态、证据或契约是否可读？
- 目标视口是否无非预期横向溢出？
- console error / warning 是否为 0？
- 截图是否处于默认阅读状态？

## 7. 计划交接

- 是否只承接已验证问题或用户明确选择的目标架构？
- 是否避免把未复现风险变成任务？
- 是否避免把场景切换、诊断卡片和筛选器当成产品 UI？
- 是否分开列出前端、后端、协议、存储、运行时和验证？
