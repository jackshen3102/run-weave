一句话: 一个终端 = 一个 run；主 Agent pane 调度，worker 是同 tmux session 里的额外 pane； 后端只做薄路由（状态守卫 / 定验收来源 / 建 pane / 注入 prompt / 消费 outbox），不替 agent 做智能决策。

1 · 生命周期状态机

一个 terminal session 同时最多绑定一个 active run。phase 是主轴，status 是当前是否卡住。

验收来源前置门：startRun 先解析 verification（planFilePath / testCaseFilePath）。没有可追溯测试案例文件时，run 停在 intake，向主 Agent 注入生成指令，等它写 docs/testing/*-test-cases.md 后回调 propose-split，才继续。
默认（前端）：autoApproveSplit=true。一旦有可追溯用例，intake 后跳过 proposal 直达 executing，无需人工确认。
proposal / need_human 这道拆分人工门只在 autoApproveSplit=false（API 直连 / CLI / 测试）时才出现。
旁路：executing 中 loop 熔断或复验超时 → need_human（冻结 worker，保留现场）→ resume(note) 清零后继续；用例生成指令注入失败或复验彻底耗尽 → failed。

3 · 核心 Loop：事件驱动，不是轮询

发条来自 worker 完成事件；后端订阅后闭环推进。串行顺序 code → code_review → behavior_verify，失败反弹回 code。

4 · 人工恢复 & 复验兜底

熔断后人工恢复 · resumeRun(note)

  • 清空 errorFingerprints + noProgressCount 归零 + escalated=false
  • 所有用例 consecutiveFail=0，bestPassCount 按当前 pass 数重算
  • note 通过 buildHumanNotePrompt 注入主 Agent 上下文
  • 恢复 active worker，loop 重新计数继续推进

复验看门狗 · recheck watchdog（每 10s）

  • 扫描已派复验但 超时 1h 未更新 outbox 的用例
  • outbox 有更新 → 直接 applyRound 收结果
  • 未超上限 → 换 fresh pane 重试（≤ 2 次）
  • 重试耗尽 → 该用例判 fail，升级 need_human

5 · 关键 loop 参数（AgentTeamLoop）

状态字段

round
验收轮次
bestPassCount
历史最高通过数 = 客观进展信号
errorFingerprints
归一化失败签名，识别“同错重复”
noProgressCount
无进展累计（熔断计数）
stableFailThreshold
默认 2 · debounce，单轮抖动不算数
maxNoProgress
默认 3 · 连续无进展熔断阈值

四个设计取舍（数字化）

  • 验收用例必须可追溯：只认 docs/testing/*-test-cases.md 里 ### CASE-000 解析出的用例（带 sourceCaseId/sourceFilePath），没有就停在 intake 让主 Agent 先写，杜绝“泛化默认用例假装验收”。
  • debounce=2：一次偶发 fail 不反弹 code，避免抖动打断改动。
  • 熔断=3 轮无进展：只有 pass 数上升 或 有 diff 才算进展，防“改了但没效果”的死循环。
  • 串行 code→review→verify + 局部重跑：保证每次验收针对最新代码；behavior_verify 默认只重跑失败/未执行/依赖 case，全量重跑要在 outbox 写明原因。
  • 熔断不删 pane/outbox：冻结只停止注入，现场留给人；符合“可见流程 + 人工验收”边界。
