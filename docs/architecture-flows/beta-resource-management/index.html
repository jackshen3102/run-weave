三个事实源，四类身份字段

lease 不是健康检查；manifest 不是实时进程表；runtime 存活也不代表它属于当前 Session。系统必须 join 后才能做安全判断。

Current architecture
1

容量事实 · Slot lease

通过 hard link 原子发布，保证 pool-01～05 同一时刻只有一个 owner。

  • slotId + leaseNonce
  • ownerSessionId + revision
  • allocatorPid 仅是申请 lease 的 CLI
2

控制事实 · Session manifest

记录 planned → starting → ready → stopping → stopped，以及失败时的 stale / failed。

  • source root + revision
  • 每个 service 的 expected identity
  • failure、cleanupStatus、recovery 指引
3

运行事实 · Runtime identity

用独立证据证明进程、锁、健康接口和 CDP 都仍属于同一 Session。

  • process signature + PID
  • Backend/App Server lock + health
  • Electron status + CDP metadata
安全操作判定 = Lease owner ∩ Manifest owner ∩ Runtime identity

三者不一致时，当前实现选择拒绝停止或释放。这避免误杀，但也意味着“身份不可证”会转化为资源池容量占用。

正常闭环与失败分叉

开始和停止都不是简单的“启动 App / 杀进程”，而是围绕 lease、身份验证和可恢复状态的事务。

Sequence

Start：先占容量，再证明组件身份

  1. Planner 选择 beta profile根据 changed paths 计算最小合法 profile。
  2. Janitor 扫描旧槽位只恢复身份和进程缺失都能安全证明的资源。
  3. 写 planned manifest固定 Session ID、source root 与 revision。
  4. 原子发布 slot lease生成 leaseNonce；dry-run snapshot 不构成租约。
  5. 更新并启动物理 BetaApp、Backend、App Server、两个 CDP 使用独立 namespace。
  6. 两轮 ownership handshakereadiness 后再次核对 lock、health、process signature。
  7. manifest → ready只有全部组件身份一致才对外返回成功。
本轮真实分支：第一次 pool-05 在 component ownership handshake 失败；安全停止、重置并释放 lease，manifest 记为 failed / leaseRetained=false。第二次申请成功。

Stop：先证明 owner，再释放容量

  1. 锁定精确 Session禁止按端口或“最近窗口”猜测目标。
  2. 校验 lease owner + nonce防止释放已被其他 Session 接管的槽位。
  3. 校验每个 dedicated servicePID 必须存活且 process signature 完全一致。
  4. 停止 owned services调用 Beta control，再停止仍属于本 Session 的进程。
  5. 确认 slot 无任何相关进程status、lock 和路径引用必须都安全。
  6. 重置 mutable state + retention清理 profile/runtime 临时状态并保留有限回滚产物。
  7. release lease + stopped删除 lease 前再次验证文件 identity。
Stale 分支：如果 process signature、lock 或 health 对不上，系统拒绝 reset / release，保留 lease 并要求 cleanup-stale。它优先保护其他现场，而不是优先释放容量。

Causal analysis

建议先建立可解释的资源控制面

推荐顺序刻意不从“自动清理”开始。没有统一事实投影时，任何更激进的释放策略都会把容量问题变成误杀风险。

Proposed direction
01

增加统一的只读 pool projection

每个 slot 同时返回 lease、manifest、runtime、recovery 四组信号，并明确 derived state 与 reason。不要再让操作者拼接多个 JSON 和本地文件。

idle→healthy/partial/stale-reclaimable/stale-manual/broken
优先级最高
02

暴露 janitor 扫描与恢复摘要

runStart() 当前 await janitor 后丢弃返回值。应让 start/status 输出每个 slot 的 active / recovered / broken 原因，并写入可追踪元数据。

低风险
03

为 ready Session 增加持续 ownership 信号

不要复用 allocatorPid。可由长期运行的 Beta component 或轻量 control heartbeat 表达 owner 活性，让“launcher 已退出”和“整组 runtime 已失效”成为两个明确状态。

需设计协议
04

只自动执行可证明安全的恢复

manifest 非活跃、lease identity 匹配、相关进程确认不存在时允许 janitor 回收；任一身份不确定仍进入 stale-manual，保留当前 fail-safe。
