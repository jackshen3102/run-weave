根因分类

已确认框架缺陷

Prompt 传输没有容量协议

只按换行切分，单个超长段落可直接进入 tmux 参数。失败发生在 worker 启动前，但 UI 已接近“投递中”语义。当前已增加 8 KiB UTF-8 分片，仍需端到端回归。

已确认框架缺陷

Dispatch 持久化与 pane 启动非原子

run 可显示 active dispatch，而 pane 仍停在旧 transcript。缺少 delivery ack 导致 UI 把“路由意图”显示成“已开始执行”。

已确认框架缺陷

Thread 恢复策略只有拒绝，没有闭环

为保护上下文而禁止新开 thread 是合理门禁；但没有 pane generation、context snapshot 或安全重建动作，因此每次都升级 Human Gate。

合同设计问题

一个 case 同时承担核心和边界行为

严格顺序执行时，边界缺陷会阻塞大量独立 case。WTC-009 是典型：核心恢复与多 Session 边界应独立优先级和阻塞策略。

环境能力问题

验收合同未在 dispatch 前做 surface 预检

直到 WTC-020 执行时才发现 app surface 不存在。该信息在 dev:open 能力表中可提前获知，应在 split/readiness 阶段发现。

已确认框架缺陷

验收源文件生命周期不稳定

run 内保留了解析结果和 SHA，但 refresh_acceptance 强依赖项目内原 Markdown；并发迁移删除文件后，用户已授权的合同调整仍无法落地。

量化结论

观察数字含义
验收完成度95.7%（22/23）实现总体已收敛，尾部不是大规模代码失败。
人工恢复密度15 次 / 14 个完成 round ≈ 1.07几乎每推进一轮就需要一次人工控制面动作，自动 loop 名义成立、恢复闭环不成立。
严格顺序的放大WTC-009 后 11 条一度全部 blocked首错即停适合单 case 内，不适合无依赖 case 的全局串行饥饿。
框架与产品故障混合至少 5 类框架/环境阻塞，2 个主要产品缺陷运行时间不能直接解释为 Code worker 效率。

后续优化建议

P0

建立 dispatch 两阶段提交：persist → deliver → ack

新增 deliveryState=requested|submitted|acknowledged|failed、operationId 和 pane generation。只有收到目标 thread 的 ack 才在 UI 显示“执行中”；超时自动撤销 active dispatch 并安全重派。

目标：消除“已抛回但 pane 未启动”；预计减少 3–4 次人工介入。
P0

Prompt 使用有界传输协议

保留 8 KiB UTF-8 分片，同时在发送前记录总字节数、chunk 数和最大 chunk；更长 prompt 改为临时文件/stdin，命令行只传引用与 SHA。加入 8 KiB、64 KiB、256 KiB 中文混合端到端用例。

目标：启动失败率归零；失败可在投递前诊断。
P0

把 thread 恢复从 Human Gate 变成显式状态机

区分 resumable、stale-but-snapshotable、lost。为 stale thread 保存 dispatch prompt、outbox baseline、repair keys 和证据引用；允许在同 pane 新 generation 恢复，而不是笼统“禁止新开 thread”。

目标：减少反复 interrupt + dispatch；预计减少 5 次以上介入。
P1

验收前置 capability negotiation

测试案例声明 requiredSurfaces；split/readiness 阶段与 Dev Session capabilities 求交集。缺失 app surface 时，在执行前就要求拆 case、skip 或选择支持该 surface 的 profile。

目标：WTC-020 类阻塞从 17 小时尾部前移到分钟级。
P1

Case 调度从“全局首错即停”改为依赖图

单 case 内仍严格遇错即停；case 之间只阻塞显式 dependsOn 后继。P1 边界失败不得饥饿无依赖 P0；自动拆分核心路径与 deferred 回归。

目标：避免 WTC-009 一次失败阻塞 11 条独立 case。
P1

协议校验在 worker 提交时同步完成

outbox 写入后先本地 schema/invariant 校验，失败时保持原 thread 可补交；不要消费 dispatch 后再发现 invariant 不一致。协议补交只更新 outbox，不重新触发产品验证。

目标：消除 Round 12–13 的补交循环。
P1

将验收合同快照变成 run 的不可变附件

split 时复制原文件到 run artifact，保存 path + SHA + parsed AST。refresh_acceptance 基于附件生成新版本；项目文档迁移不应删除运行中的事实源。

目标：用户授权后可直接调整 WTC-020，不受并发文件删除影响。
P2

UI 分开展示“已路由、已投递、已启动、已产出”

当前“已抛回 Code pane”过度承诺。显示 dispatchId、requestedAt、ackAt、lastActivityAt、outbox mtime 和阻塞层级，并为框架故障提供明确的“恢复投递”入口。

目标：用户无需靠 tmux transcript 判断真状态。
一句话结论：这次 run 证明了“严格验收”本身有价值——它确实抓到了 WTC-009 和 WTC-012；但 Agent Team 目前的自动化只覆盖正常路径，恢复路径仍依赖主 Agent 手工编排。下一阶段不应继续增加更多 gate，而应优先让已有 gate 具备可观测、可恢复、可重放的闭环。
