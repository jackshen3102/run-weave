Runweave local development isolation

用一个 Dev Session，完成从改动范围到真实验收

最好的开发方式不是手工选择一堆端口，也不是每次部署 Beta。源码侧 Planner 先推荐最小 profile，Dev Session 再明确引用默认服务或拥有独立服务，最后打开准确的 Web、Desktop 或 Terminal Browser surface。代码、脚本和开发用 Agent/Team 始终留在 Stable 主应用 terminal；所有 profile 都只是被测目标。

核心不变量

Dev Session 是任务事实源，serviceInstanceId 是运行服务事实源。Stable terminal 是唯一控制面，Dev Server、Electron Dev 和 Beta 都是被测面。未受影响的服务可以复用；受影响的服务才独占。共享不等于隐式发现。

推荐开发工作流：一个源码侧入口

以下是目标交互，当前尚未实现。自动判断只做推荐；用户目标和显式 profile 始终优先。无论选择哪种 profile，以下命令和开发 Agent/Team 都留在 Stable 主应用 terminal 中。 

pnpm dev
  └─ recommend profile + print service plan

pnpm dev --profile frontend
pnpm dev --profile fullstack
pnpm dev --profile app-server
pnpm dev --profile electron
pnpm dev --profile beta --instance agent-a

pnpm dev:status --json
pnpm dev:open --surface <web|desktop|terminal-browser>
pnpm dev:stop
01
Planchanged paths + 用户意图 + 验收 surface → 推荐最小 profile。
02
Resolve四项服务分别判为 dedicated、shared-declared 或 disabled。
03
Start只启动 dedicated 服务；共享默认 Backend/App Server 只记录引用。
04
Handshake核对 serviceInstanceId、capability、revision 和 resource namespace。
05
Open & Verify返回准确 URL/CDP surface，用真实浏览器或桌面验收。
06
Stop只清理 Session 拥有的进程；共享服务和其他 Session 不受影响。
候选方式 优点 为什么不是主方案
手工命令/文档约定 零基础设施成本 无法稳定管理身份、复用和清理
纯 diff 自动启动 操作最少 不知道用户意图、状态写入和验收目标
每次全量 Beta/沙箱 隔离最强 Frontend/Electron 日常开发反馈过慢、资源过重
Dev Session：推荐 + 覆盖 最小环境、允许共享、身份可诊断 推荐：需要统一 manifest/status/open/stop

分层判定：逐项决定服务 ownership

全局服务不是问题；未经范围判断的隐式复用才是问题。“代码未改”允许共享，但还要继续检查契约、状态和生命周期。

A
服务代码或配置改了吗？是 → dedicated。否 → 继续检查影响闭包。
B
接口、事件、认证或启动契约受影响吗？是 → provider 与关键 consumer 进入同一独立验证闭环。
C
测试会写入可能冲突的共享状态吗？是 → dedicated 或显式 namespace；不能因为代码没改就共享 profile、event log、browser session。
D
启动、停止、重连、更新或回滚是验收对象吗？是 → 本次 Dev Session 必须拥有该服务生命周期；四项都否才使用 shared-declared 默认服务。

按改动范围选环境

切换场景，观察四项服务的最小 ownership。先解决开发目标，再决定是否需要 Beta。

Dev Session Capsule：任务与服务身份分离

devSessionId 表示一次工作，serviceInstanceId 表示真实服务。端口、PID、profile 路径只是 transport/resource 字段。 

devSessionId: dev-agent-a
profile: electron
source: worktree + revision

frontend: dedicated
  serviceInstanceId: frontend:dev-agent-a
  ownerDevSessionId: dev-agent-a
backend: shared-declared
  serviceInstanceId: backend:stable-default
  resourceNamespace: stable-default-profile
appServer: shared-declared
  serviceInstanceId: app-server:stable-default
cdp.desktop: dedicated
  serviceInstanceId: cdp:dev-agent-a:desktop
cdp.terminalBrowser: dedicated
01
先决策模式根据 changed paths 和验收目标选择 frontend、full-stack、app-server、electron 或 beta。
02
生成 manifest记录 devSessionId、profile、revision、每项 serviceInstanceId、ownership 和资源边界。
03
启动或声明复用dedicated 由本 Session 启动；shared-declared 可以是默认服务，但必须记录解析后的 serviceInstanceId 和允许共享的理由。
04
逐边握手Frontend↔Backend、Backend↔App Server、Agent↔CDP 校验 service instance、capability、revision 和 namespace。
05
统一清理只停止 manifest 拥有的 PID/lock/home；stale 资源进入诊断，不按端口盲杀。

当前串线风险

事实、结构性风险和历史问题分开展示。结构性风险不等于已确认用户故障。

渐进式落地

先消灭隐式发现，再扩展多 Beta；避免用最昂贵环境验证最小前端改动。

阶段 交付 解决的问题 暂不解决
1. Dev Session Planner changed paths + intent + surface → 推荐 profile 避免所有任务默认 Beta 服务身份握手
2. Manifest/status/open/stop devSessionId + serviceInstanceId + ownership 统一事实源、可诊断 阻断错误连接
3. 禁止 ambient discovery dedicated/shared-declared/disabled 显式策略 允许有意复用默认服务，阻止无意串入 多 bundle
4. Identity handshake 每条服务边 fail closed 端口正确但实例错误 跨 revision 并行
5. 生命周期编排 统一 start/status/stop/cleanup orphan、stale lock、盲杀 动态 macOS App identity
6. Multi-Beta per-instance bundle/userData/App Server/CDP 两个 revision 同时验证 远程共享环境
上一份 Beta 多实例/CDP 计划属于第 6 阶段。更稳妥的顺序是先完成 1–5 的通用 Dev Session，再让 Beta 复用同一套 Planner、service identity、open 和 stop 协议。
