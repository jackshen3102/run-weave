# TTP-006 测试案例评审

## 结论

TTP-006 的核心不变量合理且应保留：只有 `has-session` 明确返回 exit code 1 才能判定 Session 不存在；timeout、权限错误和其它基础设施错误不得触发 thread 初始化、tmux 重建或 watcher exited。

但当前 Case 的触发动作与响应契约不明确，本次“HTTP 200 + UI running”不能判定为产品失败。应先修正 Case，再重测；本轮结果应记为测试设计/执行无效，不应阻塞 TTP-007。

## 发现

- **P1：`Backend API 或真实 Terminal UI` 不是可验证的唯一触发器。** `GET /api/terminal/session` 只投影 store，必然可以返回 200 和 `running`；`GET /api/terminal/session/:id` 在 tmux 查询失败时会回退持久化 scrollback；已有 WebSocket/runtime 也可能被直接复用。上述路径都不保证发生一次新的 runtime attach。位置：`docs/testing/terminal/terminal-tmux-persistence-and-lifecycle.testplan.yaml:155`、`backend/src/routes/terminal.ts:163`、`backend/src/terminal/runtime-launcher.ts:541`。修正方向：指定一个必经 `hasSession` 且不吞错的触发器，例如 `GET /api/terminal/session/:id/panels`，并在请求前关闭本 Case 已有 WebSocket/附着客户端。

- **P1：`明确的可重试基础设施失败` 没有定义 HTTP/WS 合约。** 当前 panels 路由会把异常映射为通用 500，WebSocket 重建失败会发送 error 后以 1011 关闭；代码里没有稳定的 `retryable` 响应字段或 tmux 暂时不可用错误码。因此执行者无法一致判断“明确”。位置：`docs/testing/terminal/terminal-tmux-persistence-and-lifecycle.testplan.yaml:143`、`backend/src/routes/terminal-panel-routes.ts:176`、`backend/src/ws/terminal-server.ts:120`。修正方向：若只验证现有行为，明确期望 panels 请求 500/WS 1011；若产品确实要求可重试协议，先定义并实现例如 HTTP 503 + `code=tmux_temporarily_unavailable` + `retryable=true`，再把它写入 Case。

- **P2：没有 timeout 日志说明本轮没有证明 Backend 执行了目标 `has-session`。** `hasSession` 对非 exit code 1 会重新抛错并记录 `terminal.tmux.has-session.failed`，底层 5 秒超时还会记录 `terminal.tmux.command.timeout`。两条日志都不存在时，不能用 HTTP 200 推断错误分类实现失效。位置：`backend/src/terminal/tmux-session-service.ts:69`、`backend/src/terminal/tmux-process.ts:225`。修正方向：把目标 socket、session name、请求起止单调时间以及两条日志设为必需证据；未观察到查询进入时应判定 fixture/触发前置条件失败。

## 推荐的最小重测合同

1. 关闭本 Case Terminal 的现有 WebSocket/附着客户端，记录 store、threadId、rebuild 和同名 Session 基线。
2. `SIGSTOP` 目标 tmux server，并用同一 tmux binary、socket、session name 证明直接 `has-session` 在约 5 秒超时且不是 exit code 1。
3. 请求 `GET /api/terminal/session/:id/panels`，确认发生同一目标的 `terminal.tmux.command.timeout` 与 `terminal.tmux.has-session.failed`；按选定合同断言通用 500，或先实现并断言结构化 503 retryable 响应。
4. 单独读取 `GET /api/terminal/session` 或 store，预期仍为 `running`；这项是“未误判 Session 丢失”的正向证据，不是失败证据。
5. 确认没有 session-missing、agent-resume、rebuild 增量、替代 Session 或 watcher exited。
6. `SIGCONT` 后重新附着，确认同一 PID、Socket、Session、threadId 和 marker 恢复。

## 更简单的替代方向

如果本轮只验证“非 exit code 1 不会重建”，无需把 UI 暂时错误状态纳入 TTP-006。固定使用 panels API 触发 `hasSession`，以日志、store 和 tmux 身份取证；UI 的结构化 retryable 提示可拆成独立 Case。这样可以避免 API fallback、WebSocket 复用和 UI 状态投影造成的歧义。
