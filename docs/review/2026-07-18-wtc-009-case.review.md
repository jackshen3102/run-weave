# WTC-009 用例合理性评审

## 结论

WTC-009 的验收目标有来源，当前失败也是真实产品行为，不应按误报跳过。计划明确要求按生效 Project 恢复 Terminal 与 Preview 状态，并要求刷新后恢复选择；用例把它具体化为恢复 Preview path。真实证据显示父 Project、子 Project 和 Terminal 均已恢复，只有 Preview 从 `Open/context.txt` 复位为 `Changes/path=null`。

不过，该用例把“父/子 Project 往返切换恢复”和“整页刷新后恢复”两个生命周期合成一个 P0，诊断粒度不够好；建议后续拆成两个独立 case，并单独确认刷新恢复 Preview path 的严重级别。

## 发现

- **P1 严重：当前 WTC-009 fail 有效，不能降为 skipped。** 冻结规则要求切换列表项后恢复该生效 Project 的 Preview 状态，验证章节又要求刷新后恢复选择；当前测试计划明确把 Preview path 写入期望与失败判定。当前 run 必须按已冻结 acceptance 执行，不能在失败后收窄解释。定位：`docs/plans/2026-07-18-worktree-terminal-context.md:46`、`:331`；`docs/testing/terminal/worktree-terminal-context.testplan.yaml` 的 WTC-009。
- **P2 一般：WTC-009 混合了两个独立状态生命周期。** 证据表明父 Project/context 往返时 Preview path 能正确恢复，只有浏览器刷新后丢失；一个总 case 失败会掩盖前半部分已经通过。建议拆为“跨父/子切换独立恢复”和“页面刷新恢复 Preview path”两个 case。定位：`docs/testing/terminal/worktree-terminal-context.testplan.yaml` 的 WTC-009。
- **P2 一般：刷新恢复 Preview path 的 P0 等级存在文档歧义。** Workspace 设计只明确列出 `contextProjectIdByParentProjectId` 与 `projectSessionIds` 的持久化，Preview 部分只说明按 `projectId` 分桶；“刷新后恢复选择”没有逐字段定义。当前 case 已经消除了本次 run 的执行歧义，但后续计划应明确 `mode/path/selectedFilePath/selectedChangePath` 中哪些字段跨刷新持久化，并决定是 P0 还是 P1。定位：`docs/plans/2026-07-18-worktree-terminal-context.md:190`、`:193`、`:327`、`:331`。
- **P2 一般：Round 6 的持久化修复需要保持连接作用域。** 当前工作树把 Preview projects 持久化到固定 localStorage key；而 recent selection 使用带 `apiBase` 的 key。若最终方案继续使用全局 key，多个 backend/profile 复用同一 Project ID 时可能串 Preview path。建议只持久化稳定的 Preview 选择字段，并按连接 scope 隔离。定位：`frontend/src/features/terminal/preview-store.ts:31`、`:294`；`frontend/src/features/terminal/recent-selection.ts:14`。

## 更简单的用例结构

1. `WTC-009A [P0]`：在两个父 Project/context 间往返，分别恢复 effective Project、Terminal 与 Preview path；不刷新页面。
2. `WTC-009B [P1]`：刷新页面后恢复当前 effective Project、Terminal 与 Preview mode/path；明确需要持久化的字段和 storage scope。

这样可以保留核心隔离要求，同时让失败直接指向内存切换恢复或跨刷新持久化，不必用一个 P0 覆盖两个不同故障域。
