# Terminal Browser 页面评论原型

右侧 Terminal Browser “注释 / Browser comments”功能的系统梳理与可运行交互原型。原型只使用 mock 数据，不修改或连接真实 Electron、BrowserView、终端和 Agent。

## 启动

```bash
python3 -m http.server 6189 --directory docs/prototypes/terminal-browser-annotations
```

可直接进入不同状态：

```text
http://127.0.0.1:6189/?state=idle
http://127.0.0.1:6189/?state=selecting
http://127.0.0.1:6189/?state=review
http://127.0.0.1:6189/?state=review&fail=1
http://127.0.0.1:6189/?state=error
```

## 优化前流程

```text
更多菜单“Add browser comments”
  → Electron 向当前 BrowserView 注入 annotation runtime
  → hover 页面元素并点击锁定目标
  → 单行输入框填写评论
  → “添加 / ⌘Enter”仅保存；“发送 / Enter / 箭头”请求立即提交整批
  → React 每 500ms 轮询 runtime 状态
  → Electron 读取全部评论、截取一张整页图、停止 runtime 并清空页面标记
  → Renderer 保存截图、构造 # Browser comments prompt
  → 以 prompt_paste 发送到当前 Agent 终端
```

当前数据合同只包含评论文本、页面 URL、目标文本、selector/path、位置、viewport/rect/DPR，以及整批共用的一张截图。没有草稿持久化、发送事务状态、评论列表、分类、优先级、回复或解决状态。

## 核心问题清单

| 级别 | 当前问题                         | 代码事实 / 触发条件                                                          | 用户影响                                               | 原型中的处理                                             |
| ---- | -------------------------------- | ---------------------------------------------------------------------------- | ------------------------------------------------------ | -------------------------------------------------------- |
| P0   | 发送失败会丢失评论               | Electron `submit` 在截图后立即 `stop`；Renderer 随后才保存截图并调用终端输入 | 截图上传或终端发送失败时，只剩错误横幅，原草稿无法重试 | 失败状态保留整批草稿，明确提供“重试发送”                 |
| P1   | “添加”和“发送”边界反直觉         | 普通 Enter、箭头和“发送”会提交整批；`⌘Enter` 和“添加”才保存单条              | 用户写第一条时可能意外把整批提前发送                   | 浮层只提供“添加评论”；整批发送只存在于评论面板           |
| P1   | 缺少批量审阅与管理视图           | 页面中只有编号 marker 和单条 hover 预览                                      | 评论多时无法快速扫描、定位、编辑、删除和确认发送范围   | 右侧面板集中列出所有评论，marker 与列表双向定位          |
| P1   | 页面离开会静默丢草稿             | 主导航、页内导航、切换 Browser tab 都会停止 runtime 并清空状态               | SPA 跳转、刷新或误切 tab 都会让工作消失                | 保留为后续阶段，不纳入本次第一阶段                       |
| P1   | 评论模式劫持网页点击但状态提示弱 | runtime 捕获 document click 并阻止网页默认行为；外层主要依赖图标高亮         | 用户可能不知道为什么页面不能正常操作，也不清楚如何退出 | 持久“评论模式”状态条说明网页交互被暂停，并提供“完成选择” |
| P2   | 单行输入不适合长评论             | 输入控件为 `input`，合同允许最多 4000 字符                                   | 长说明难阅读、难编辑；Enter 又承担高风险发送           | 使用多行 textarea；普通 Enter 换行，`⌘Enter` 仅添加单条  |
| P2   | 入口与提交动作分散               | 无评论时入口在更多菜单；有评论后图标提升；提交仍在原生更多菜单               | 用户难以形成“进入—收集—审阅—发送”的稳定心智            | 保留动态入口规则，但有草稿后提升的图标直接打开评论面板   |
| P2   | 标记位置是一次性坐标             | 创建评论时保存 rect / nodePosition，runtime 未监听布局与滚动重算目标         | 页面异步布局变化后 marker 可能与目标错位               | 原型用目标身份驱动定位；实现时应在可见时按元素重新测量   |
| P2   | 一张截图承载整批评论             | 提交时只对当前可见 BrowserView `capturePage()` 一次                          | 长页面或跨可视区域的评论与截图难对应                   | 面板显式按评论组织；实现阶段需评估整页快照或逐评论裁剪   |
| P2   | 只有全局错误横幅                 | 注释错误与代理、请求头、页面错误共用横幅区域                                 | 用户不知道评论是否仍被保留、下一步该做什么             | 错误反馈靠近发送按钮，并明确“草稿仍保留”与重试动作       |
| P3   | 动态页面目标稳定性不足           | selector 依赖 DOM 路径和 `nth-of-type`，导航后状态直接清空                   | SPA 重渲染后 selector 可能失效，Agent 定位可信度下降   | 原型展示稳定的目标摘要；实现需引入重新解析与失效状态     |

## 目标流程

```text
未开启
  → 从更多菜单进入“评论模式”
  → 点击元素
  → 多行编辑单条评论
  → 添加到草稿（不发送）
  → 在右侧面板审阅 / 定位 / 编辑 / 删除
  → 明确点击“发送 N 条给 Agent”
  → 发送中：冻结本批次但不删除本地草稿
  → 成功：归档并清理；失败：保留原批次并允许重试
```

### 状态与退出规则

| 状态     | 页面点击     | 草稿       | 允许动作                             |
| -------- | ------------ | ---------- | ------------------------------------ |
| 未开启   | 正常网页交互 | 无         | 从更多菜单进入评论模式               |
| 选择目标 | 变为选择元素 | 可有       | 添加评论、完成选择、打开面板         |
| 编写单条 | 目标锁定     | 未保存     | 取消、`⌘Enter` 添加、点击“添加评论”  |
| 批量审阅 | 正常网页交互 | 保留       | 定位、编辑、删除、继续添加、批量发送 |
| 发送中   | 正常网页交互 | 冻结保留   | 等待结果                             |
| 失败     | 正常网页交互 | 原样保留   | 重试、继续编辑、放弃                 |
| 成功     | 正常网页交互 | 成功后清理 | 回到未开启                           |

## 原型验收路径

1. `?state=idle`：从更多菜单进入评论模式，评论面板为空。
2. 点击页面的 Team seats 或 Invoice email，输入多行评论；普通 Enter 换行，`⌘Enter` 或“添加评论”只新增草稿。
3. 从 marker 或右侧列表双向定位；编辑、删除后编号保持连续。
4. `?state=review&fail=1`：首次发送失败，评论仍在；再次点击重试后成功并清空。

## 功能分类

### 产品核心功能

| 元素 / 行为           | 最终产品是否需要 | 价值                         |
| --------------------- | ---------------- | ---------------------------- |
| 评论模式状态条        | 是               | 明确页面点击模式和退出入口   |
| 多行单条编辑器        | 是               | 将“添加草稿”与“发送整批”分离 |
| 评论审阅面板          | 是               | 集中定位、编辑、删除和发送   |
| marker 与列表双向定位 | 是               | 支撑多评论场景               |
| 失败保留与重试        | 是               | 让发送成为可恢复事务         |
| 动态入口规则          | 是               | 延续既有工具栏密度设计       |

### 后续阶段，不在本次落地范围

- 有草稿时的导航 / Browser tab 切换保护。
- 页面布局变化后的 marker 重新定位。
- 整页截图或逐评论截图增强。
- Agent 状态映射、操作录制、Before / After、协作评论能力。

### 原型辅助功能

## 第一阶段落地边界

- Electron 提交改成“读取快照但不销毁”，只有终端输入确认成功后再显式 clear；失败不清状态。
- Renderer 增加评论面板，继续复用现有共享 `TerminalBrowserAnnotationDraft`，不复制评论状态。
- BrowserView 编辑器改为多行输入，普通 Enter 只换行；页面内动作只添加单条草稿。
- `active` 与 `selecting` 分离：完成选择后保留 marker 和草稿，恢复正常网页点击。
- 分类、优先级、回复、解决状态暂不进入第一版。当前目标是把“页面证据可靠送给 Agent”的主流程做完整，而不是先扩成通用协作评论系统。
